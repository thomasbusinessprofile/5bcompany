## CRITICAL

### C1. Hardcoded admin password committed to repository
- **File:** `supabase/migrations/0013_bootstrap_admin.sql:18,19`
- **Detail:** Plaintext password `ZJTxUx3PK2AmXXk!Vr@E` and `Admin5b!2026` (from `execute_sql` reset earlier) is in git history.
- **Risk:** Anyone with read access to the repo (currently private but the secret is forever in history) can reuse it. Especially bad if repo is ever made public or backed up off-platform.
- **Fix:**
  1. Rotate password now via Supabase Studio → Authentication → Users.
  2. Replace the literal password in `0013_bootstrap_admin.sql` with `'change-me-before-running'` and add a header comment requiring rotation.
  3. Consider `git filter-repo` to scrub history if repo will ever be public.

### C2. Missing `resend` package dependency
- **File:** `app/lib/email/resend.ts`, `package.json`
- **Detail:** Email send uses raw `fetch` to `api.resend.com`. **This is actually fine** — no SDK required (verify: it's a single POST). Marking as **NOT A REAL FINDING** unless we adopt the SDK. Leaving here for the record.

### C3. Open-redirect via `redirect_to` form param
- **Files:** `app/admin/pipeline/actions.ts:52,67`, `app/admin/activities/actions.ts:55,72`, `app/admin/contacts/actions.ts`, etc.
- **Detail:** `val(formData, "redirect_to")` is passed straight to `redirect()`. An attacker who can get a logged-in admin to submit a form on a hostile page (CSRF mitigated by Next 15, but trusted-CSRF or social-engineered form on partner site) can redirect them to an attacker domain.
- **Fix:** Normalise to internal paths only:
  ```ts
  const redirectTo = val(formData, "redirect_to");
  const safe = redirectTo.startsWith("/") && !redirectTo.startsWith("//")
    ? redirectTo
    : null;
  redirect(safe || "/admin/...");
  ```
  Also apply to `next` param in `login/actions.ts:47`.

## HIGH

### H1. Server actions don't re-check role server-side
- **Files:** all `app/admin/*/actions.ts` — saveProduct, saveArticle, saveContact, saveCompany, saveDeal, saveActivity, deleteX, sendComposedEmail, moveDealStage, etc.
- **Detail:** Middleware blocks page navigation. **But server actions are POST endpoints addressable by their action hash; middleware does not run for them in all cases.** RLS catches DB writes if the user has no admin role, but mutations that don't go through Supabase RLS-checked tables (e.g. revalidatePath, redirect) could still produce side effects.
- **Fix:** Top of every admin action:
  ```ts
  const { data: { user } } = await supabase.auth.getUser();
  const { data: profile } = await supabase.from("profiles").select("role").eq("user_id", user?.id).maybeSingle();
  if (!canAccessAdminArea(profile?.role)) redirect("/login?status=unauthorized");
  ```
  Extract to `requireAdminRole(supabase)` helper.

### H2. Email `from` header user-controlled
- **File:** `app/admin/email/actions.ts:81`
- **Detail:** `from` comes from FormData. Email header injection (CRLF) risk; spoofing the sender domain.
- **Fix:** Whitelist or hardcode `from` from env (`RESEND_FROM_EMAIL`). Reject any value containing CR/LF.

### H3. RLS policy `inquiries public insert` always-true on WITH CHECK
- **Source:** Supabase advisor `rls_policy_always_true`
- **Detail:** `WITH CHECK (true)` for INSERT on `inquiries` allows any anonymous user to insert arbitrary inquiry rows. Currently used by the public RFQ form (intentional) but no rate-limit, no input cap → spam vector.
- **Fix:** Keep INSERT public but add per-IP rate-limiting at the edge (middleware or Vercel rate-limit), and add a Postgres trigger that rejects inserts where `length(message) > 5000` or `length(email) > 254`.

### H4. Leaked-password protection disabled
- **Source:** Supabase advisor `auth_leaked_password_protection`
- **Detail:** Supabase Auth can check passwords against HaveIBeenPwned. Currently OFF — users (including admin) can register with passwords from known breaches.
- **Fix:** Supabase Studio → Authentication → Policies → enable "Leaked password protection". 1 click.

### H5. No rate limit on outbound email
- **File:** `app/admin/email/actions.ts`
- **Detail:** Any admin can fire `sendComposedEmail` repeatedly. Resend free tier = 3 k/month. Compromised admin token or buggy automation could exhaust quota in minutes.
- **Fix:** Insert a row check on `crm_emails` with `created_at > now() - interval '1 hour'`; if count > N, reject. Or use Upstash Rate Limit.

## MEDIUM

### M1. `auth_rls_initplan` re-evaluating `auth.uid()` per row
- **Tables:** `profiles` (3 policies)
- **Fix:** Migration to wrap with subselect:
  ```sql
  alter policy "profiles read own or admin-like" on public.profiles
  using ((user_id = (select auth.uid())) or app_private.is_admin_like());
  ```
  Repeat for the other two profile policies. Same pattern can apply to any later policy using `auth.uid()`.

### M2. Multiple permissive policies on `sourcing_requests` UPDATE
- **Source:** advisor `multiple_permissive_policies`
- **Detail:** Buyer-update policy and staff-update policy both fire for every UPDATE row, doubling RLS cost.
- **Fix:** Combine into a single policy with `OR` between conditions, or use one as `restrictive`.

### M3. Email + URL not validated in compose action
- **File:** `app/admin/email/actions.ts:66`
- **Fix:** Run `to`/`cc`/`bcc` through `validateEmail()` (already exists in `app/lib/form-utils.ts`).

### M4. Date parsing without format validation
- **Files:** `app/admin/articles/actions.ts:37`, `app/admin/activities/actions.ts:36`
- **Fix:** Validate ISO 8601 with regex before `new Date()`.

### M5. `next` redirect param in `/login` not normalised
- **File:** `app/login/actions.ts:47`
- See C3 — same class.

### M6. Console.error in production paths
- **Files:** `app/error.tsx`, `app/shared/analytics.ts:30`
- **Fix:** Replace with a `log()` wrapper that no-ops in production or sends to Vercel logs only.

### M7. Unbounded SELECTs in CMS server data
- **Files:** `getProductCmsData`, `getCmsArticles`, `listContacts`, `listCompanies`, `listDeals`
- **Detail:** No `.limit()` on most CMS queries. Fine at current scale (≤ 30 products, ≤ 100 articles) but unbounded growth = full table scan on every page render.
- **Fix:** Add `.limit(500)` defensively; add pagination UI later.

### M8. Hardcoded session ID in commit messages
- **Files:** Every commit message
- **Detail:** Each commit ends with a `https://claude.ai/code/session_...` URL. Internal-only. Harmless but leaks tool usage to a public repo if it goes public.
- **Fix:** Optional — strip in pre-commit hook.

## LOW

### L1–L10. Unused indexes (10 of them)
- **Source:** advisor `unused_index`
- **Detail:** Indexes on `inquiries_product_id_idx`, `products_category_id_idx`, `quotations_*`, `request_*`, `sourcing_requests_*`, `articles_*`, `crm_*` have never been used.
- **Fix:** Most are correctly placed for future query patterns — keep. Drop only after 6 months of zero usage. Safe to ignore for now.

### L11. Unindexed FKs (10 CRM tables)
- **Source:** advisor `unindexed_foreign_keys`
- **Detail:** `owner_id`, `inquiry_id`, `activity_id`, `template_id`, `tag_id` FK columns on CRM tables lack covering indexes. Cascades on DELETE will full-scan.
- **Fix:** One migration adding `create index crm_<table>_<fk>_idx on public.crm_<table>(<fk>);` for each. Cheap.

### L12. Commit messages leak Claude session URL
- See M8.
